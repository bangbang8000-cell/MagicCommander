## 一、项目基础信息梳理总结

### 1. 业务类型
**工具类桌面软件** - 网络设备配置管理工具，面向网络工程师和运维人员，用于批量生成网络设备配置文件。

### 2. 核心业务功能清单
| 功能模块 | 功能描述 | 实现状态 |
|---------|---------|---------|
| 项目管理 | 创建、删除、切换项目，目录结构自动生成 | ✅ 已实现 |
| 模板编辑器 | Jinja2 模板文件编辑，语法高亮 | ✅ 已实现 |
| Excel 管理 | 参数表查看和编辑，支持 .xlsx/.xls | ✅ 已实现 |
| 文件查看器 | 文本/Word/Excel 文件查看 | ✅ 已实现 |
| 配置渲染 | 基于 Jinja2 + Excel 批量生成配置 | ✅ 已实现 |
| 标签生成 | 设备标签文档生成（Word 格式） | ✅ 已实现 |
| 状态持久化 | 项目状态、编辑器状态、UI 状态保存 | ✅ 已实现 |

### 3. 原型完成程度
**完整功能实现** - 已完成前后端联调，具备完整的项目管理、文件编辑、配置渲染功能，可直接用于生产环境。

### 4. 技术栈明细
| 层级 | 技术 | 版本 |
|-----|-----|-----|
| 前端框架 | React + TypeScript | 18.x / 5.x |
| 构建工具 | Vite | 5.x |
| 样式方案 | TailwindCSS | 3.x |
| 状态管理 | Zustand + persist | 4.x |
| 代码编辑器 | Monaco Editor | 4.x |
| 主进程 | Electron | 28.x |
| IPC 安全 | contextBridge + preload | ✅ |
| 后端脚本 | Python + Jinja2 | 3.x |
| Excel 处理 | xlsx (Node) + openpyxl (Python) | ✅ |
| Word 解析 | mammoth | 1.x |

### 5. 当前已知痛点
| 痛点 | 状态 | 备注 |
|-----|-----|-----|
| 文件显示问题 | ✅ 已修复 | 编辑器容器高度为 0 导致 |
| 浅色模式配色不协调 | ✅ 已修复 | 已优化各组件配色 |
| Python 进程管理 | ⚠️ 待优化 | 进程启动/销毁逻辑可改进 |
| 异常处理不完善 | ⚠️ 待优化 | 缺少统一错误处理机制 |
| 国际化支持 | ❌ 未实现 | 仅支持中文 |

### 6. 目标用户群体
- **主要用户**：网络工程师、运维人员
- **使用场景**：批量配置网络设备（交换机、路由器等）
- **技能水平**：熟悉网络设备配置，了解 Jinja2 模板语法

### 7. 业务特殊约束
| 约束项 | 说明 |
|-----|-----|
| 运行环境 | 桌面应用，需安装 Python 环境 |
| 数据存储 | 本地文件系统，无云端依赖 |
| 付费模式 | 无付费功能，纯工具软件 |
| 并发要求 | 单用户单进程，无并发需求 |
| 上线时间 | 可立即发布使用 |

### 角色 1：系统架构师视角

#### ① 原型现存缺陷盘点

| 缺陷编号 | 缺陷描述 | 影响 |
|---------|---------|-----|
| ARCH-001 | **前后端耦合度高**：Electron 主进程直接调用 Python 脚本，缺少中间抽象层 | 扩展困难，无法替换后端实现 |
| ARCH-002 | **数据存储分散**：项目数据在文件系统，状态数据在 localStorage，缺少统一数据层 | 数据一致性风险 |
| ARCH-003 | **IPC 接口设计不规范**：handlers.ts 中混杂项目管理和文件操作，职责不清晰 | 维护困难，代码冗余 |
| ARCH-004 | **Python 进程生命周期管理简陋**：仅用 spawn 启动，缺少健康检查和自动重启 | 进程崩溃后无法恢复 |
| ARCH-005 | **缺少模块边界定义**：前端组件、Store、服务层边界模糊 | 代码职责混乱 |
| ARCH-006 | **无配置中心设计**：路径配置散落在各文件中 | 环境切换困难 |
| ARCH-007 | **缺少性能监控机制**：无渲染性能、内存占用监控 | 性能问题难以定位 |

#### ② 分短期/中期/长期优化改进方案

**短期（1-3 天）**：
- [ARCH-S1] 统一路径配置：创建 `config.ts` 集中管理 BACKEND_DIR、用户数据目录等路径
- [ARCH-S2] 添加 Python 进程健康检查：在 PythonService 中添加心跳检测和自动重启逻辑
- [ARCH-S3] IPC 接口分类整理：将 handlers.ts 拆分为 project.handler.ts、file.handler.ts、render.handler.ts

**中期（一周内）**：
- [ARCH-M1] 设计数据访问抽象层：创建 DataService 接口，统一文件系统和状态存储访问
- [ARCH-M2] 实现 IPC 接口版本管理：为 preload API 添加版本号，支持向后兼容
- [ARCH-M3] 添加性能监控埋点：在关键操作（渲染、文件读取）添加耗时统计

**长期（二期迭代）**：
- [ARCH-L1] 设计插件化架构：支持自定义模板引擎、自定义输出格式
- [ARCH-L2] 实现多后端支持：可配置使用 Python 或 Node.js 后端
- [ARCH-L3] 设计云端同步方案：支持项目配置云端备份（可选功能）

#### ③ 落地实施优先级

| 优先级 | 方案编号 | 说明 |
|-------|---------|-----|
| **高** | ARCH-S1, ARCH-S2 | 影响系统稳定性，需立即处理 |
| **中** | ARCH-S3, ARCH-M1, ARCH-M2 | 影响可维护性，一周内完成 |
| **低** | ARCH-M3, ARCH-L1, ARCH-L2, ARCH-L3 | 性能优化和扩展功能，二期规划 |

| SEC-001 | **路径拼接存在风险**：未对用户输入的文件路径进行严格校验 | 可能访问敏感文件 |
| SEC-002 | **缺少文件类型校验**：用户可尝试打开任意类型文件 | 可能导致解析错误或安全风险 |
| SEC-003 | **IPC 接口无权限控制**：所有接口均可被渲染进程调用 | 潜在越权风险 |
| SEC-004 | **敏感路径暴露**：backend 路径可通过 app:getPath 获取 | 信息泄露 |
| SEC-005 | **缺少输入长度限制**：文件名、项目名无长度限制 | 可能导致缓冲区问题 |
| SEC-006 | **Python 命令注入风险**：项目名直接传递给 Python，未做转义 | 命令注入风险 |
| SEC-007 | **缺少崩溃日志保护**：崩溃日志可能包含敏感信息 | 信息泄露 |
| SEC-008 | **sandbox: false 配置**：Electron 禁用了沙箱 | 安全性降低 |

#### ② 分短期/中期/长期优化改进方案

**短期（1-3 天）**：

- [SEC-S2] 添加文件类型白名单：只允许打开指定类型的文件
- [SEC-S3] 添加输入长度限制：限制项目名、文件名最大长度

**中期（一周内）**：
- [SEC-M1] 实现 IPC 权限控制：为敏感接口添加权限校验
- [SEC-M2] 转义 Python 命令参数：对传递给 Python 的参数进行转义
- [SEC-M3] 保护敏感路径：限制 app:getPath 返回的路径范围

**长期（二期迭代）**：

- [SEC-L2] 实现安全审计日志：记录敏感操作日志
